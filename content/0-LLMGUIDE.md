# LLMGUIDE — LocalLLM 시리즈 작성 가이드

> 이 문서는 Claude(또는 다른 LLM)가 이 폴더에서 글을 쓰거나 편집할 때 톤, 구조, 컨벤션을 따라가도록 만든 메타 가이드입니다. 새 글을 만들거나 기존 글을 고치기 전에 먼저 이걸 읽으세요.

---

## 1. 프로젝트 목표
- **무엇**: M4 Max 맥북에서 로컬 LLM을 Claude Code 대용으로 굴리기 위한 가이드 시리즈
- **누구를 위해**: 한국 중학생도 따라올 수 있는 수준의 입문자
- **언어**: 한국어 (기술 용어는 영어 그대로 + 한글 설명 병기)
- **형식**: Obsidian 마크다운, 시리즈 형태 (1편, 2편, ...)

### 현재 시리즈 구성
- `1-사전-준비` — WezTerm + Homebrew 등 환경 셋업
- `2-llama-cpp-설치` — llama.cpp 설치
- `3-llama-cpp-vs-mlx` — 추론 엔진 비교
- `4-model-name` — 로컬 모델 이름 읽는 법
- `5-model-speed` — 속도 결정 요인
- `6-model-pick` — Qwen 3.6 27B MTP 픽
- `7-model-run` — 빌드·다운로드·실행
- `8-terminal-agent` — Pi 셋업 + 연결 + 속도 향상
- `9-qwen-35b-a3b` — 35B A3B 추가 (듀얼 모델)
- `99-community-resources` — 커뮤니티 리소스 (별도 대분류)

## 2. 폴더 구조

```
LocalLLM/
├── 0-LLMGUIDE.md              ← 이 문서 (메타 가이드)
├── 1-사전-준비/
│   ├── index.md               ← 가이드 본문. primitives를 ![[...]]로 embed
│   ├── assets/                ← 이 가이드에서 쓰는 스크린샷
│   └── primitives/            ← 도구/개념별 재사용 페이지 (한 토픽 = 한 파일)
│       ├── wezterm.md
│       └── homebrew.md
├── 2-XXX/                     ← 다음 편들도 같은 구조로
│   ├── index.md
│   ├── assets/
│   └── primitives/
└── 99-community-resources/    ← 별도 대분류 (편 시리즈와 분리)
```

### 핵심 원칙
- **`index.md`는 흐름**, **`primitives/*.md`는 단일 도구의 정본(source of truth)**
- 도구가 업데이트되면 해당 primitive 파일만 고침. index는 그대로 유지
- 같은 도구를 여러 편에서 쓸 거면 그 도구의 primitive를 공용 위치(예: 상위 폴더)로 올려도 됨. 단, 일단 기본은 각 편 안에 둠
- `assets/`와 `primitives/`는 모든 편에서 동일한 이름 사용 (다른 이름 금지)

## 3. 톤 & 보이스

### 존댓말 — 해요체 + 합쇼체 섞어 쓰기
- ~요, ~예요, ~이에요, ~돼요 (해요체)
- ~다, ~ㅂ니다, ~입니다 (합쇼체)
- 두 어미를 자연스럽게 섞어서 너무 딱딱하지도, 너무 가볍지도 않게
- 반말(~야, ~지) 절대 금지

### 중학생도 이해할 수 있게
- **전문 용어는 비유로 풀어줌**. 예시:
  - 터미널 → "검은 화면에 글씨 쳐서 컴퓨터한테 명령 내리는 창"
  - Homebrew → "맥북 전용 앱스토어 같은 거"
  - 패키지 매니저 → "도구를 명령어 한 줄로 깔아주는 도구"
- **짧은 문장**. 한 문장에 한 개념.
- **안심 멘트 자주**: "지금은 몰라도 됩니다", "그냥 깔아만 놔도 잘 돌아가니까 걱정 안 해도 돼요"
- **영어 기술 용어**(brew, WezTerm, xcode-select 등)는 그대로 쓰되, 처음 등장할 때 한글로 설명 추가
- **이모지는 절제해서**. 성공 표시(🎉), 경고(⚠️) 정도만

## 4. Primitive 페이지 구조 (모든 도구 페이지 공통)

```markdown
---
tags: [tool, <category>]      ← terminal, package-manager, llm 등
last_updated: YYYY-MM-DD
---

# <도구 이름>

## 이게 뭐예요?
도구 한두 문장 설명. 비유 사용.

## 왜 이걸 쓰나요?
- 장점 1
- 장점 2
- 단점 (있으면)
- 안심 멘트

## 설치하는 법 (or 사용하는 법)
1. 단계
   ![[스크린샷]]
2. 단계
   ...

## 잘 깔렸는지 확인              ← 설치류일 때만
명령어 + 결과 스크린샷

## 에러가 났을 때                ← 자주 발생하는 에러만
### <에러 이름>
원인 + 해결 방법 단계별로

## 알아두면 좋은 명령어          ← CLI 도구일 때만
| 명령어 | 무슨 뜻? |
|---|---|
| ... | ... |

## 더 알아보기
- 공식 문서 링크
- 설정 파일 위치 등
```

섹션은 **위 순서를 유지**. 해당 섹션이 필요 없으면 그냥 빼면 됨 (빈 섹션 두지 말 것).

---

## 5. Index 페이지 구조 (각 편의 본문)

```markdown
# <편 제목> (N편)

> 이 편이 뭘 다루는지 한두 줄로.

## 하드웨어 (또는 사전 준비)
- 사양 또는 필요한 것

---

## 1. <섹션 이름> — <도구 이름>
![[도구이름]]

---

## 2. <섹션 이름> — <도구 이름>
![[도구이름]]
```

- `index.md`는 **얇게 유지**. 본문은 primitive에 두고 여기서는 흐름만 잡음
- `---`로 큰 섹션 구분
- 각 섹션 헤더에 도구 이름 같이 넣기 (예: `## 1. 터미널 — WezTerm`)

---

## 6. 마크다운 컨벤션

### 코드 블록
- 셸 명령어는 ` ```sh ` 사용
- 인라인은 백틱 (`brew --version`)

### 콜아웃 (중요/경고)
> ⚠️ **중요!** 본문...

비밀번호 안 보이는 것 같은 헷갈리기 쉬운 부분, 데이터 손실 위험 등에만 사용. 남발 금지.

### 스크린샷 임베드
- `![[파일이름.jpg]]` 형식
- 임베드 **위에** 한 줄 설명 (밑이 아니라 위)
- 캡션은 명령형이 아니라 설명형: "다운로드 버튼이 보입니다" (O), "다운로드 버튼을 누르세요" (X — 다음 줄에서 설명)

### 표
명령어 목록처럼 짝-짝 매칭되는 정보에 사용. 텍스트로 충분하면 표 만들지 말 것.

### 링크
- 외부 URL: 그대로 노출 (`https://brew.sh/`)
- 내부 링크: `[[파일이름]]` 또는 `[[파일이름#섹션]]`
- 임베드: `![[파일이름]]`

---

## 7. 자주 빠지는 함정

- **반말로 작성하지 않기**. 존댓말 유지.
- **새 primitive 만들 때 위 템플릿 따르기**. 임의로 섹션 순서 바꾸지 말 것.
- **`tools/`, `partials/`, `_modules/` 같은 이름 쓰지 말기**. 이 프로젝트는 `primitives/`로 통일.
- **`index.md`에 본문 길게 쓰지 말기**. 본문은 primitive로 빼고 여기서는 embed만.
- **screenshot 옆에 또 다른 screenshot 붙이지 말기**. 각 단계마다 한 줄 설명 끼워넣기.
- **존재하지 않는 명령어/플래그 만들어내지 말기**. 모르면 공식 문서 링크만 걸고 빈 자리로 둠.
- **토픽 벗어나지 말기**. 모델 다운로드 primitive에 듀얼 운영 메모리 계산이 들어가지 않음. 다운로드 토픽이면 다운로드만, 운영 토픽이면 운영만. 인접 주제는 별도 primitive에서 다루고 `![[...]]`로 연결.
- **"왜 이걸 골랐는가"를 빼먹지 말기**. 모델·도구·플래그 선택할 때마다 **결정 이유**를 한 문단이라도 쓸 것. 단순 "이거 받으세요" 식 명령조는 금지. 다른 선택지가 있는데 왜 이걸 골랐는지를 설명.
- **쓸데없는 정보 적지 않기**. "더 알아보기" 같은 채움용 섹션, 일반론, 사용자가 안 물어본 부가 설명은 빼기. 사용자가 이미 안다고 가정하지 말고, 동시에 모른다고 가정해서 과잉 설명도 하지 말기. **결정·실행·검증에 직접 필요한 것만.**

## 8. 새 편/도구 추가할 때 체크리스트

- [ ] 폴더 만들기: `N-제목/`
- [ ] 안에 `index.md`, `assets/`, `primitives/` 만들기
- [ ] 새 도구/개념마다 `primitives/<이름>.md` 만들고 위 템플릿 따라 작성
- [ ] frontmatter (tags, last_updated) 빠뜨리지 않기
- [ ] `index.md`에서 `![[이름]]`로 embed
- [ ] 톤 점검: 존댓말? 비유? 안심 멘트?
- [ ] 스크린샷마다 위에 한 줄 설명 있는지 확인
- [ ] **각 primitive가 자기 토픽에만 집중하는지 확인** — 다른 primitive 영역 침범 X
- [ ] **선택·결정 지점마다 "왜"가 적혀 있는지 확인**
- [ ] 군더더기·일반론·"참고로..." 류 사족 정리
