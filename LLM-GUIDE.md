# LLM Guide — AI 한테 토스하실 때 쓰는 가이드

요새 누가 글 한 줄 한 줄 손으로 쓰시나요? 어차피 Claude Code, ChatGPT, Cursor 같은 LLM 으로 쓰시잖아요.

이 문서는 그 LLM 한테 통째로 던지라고 만든 가이드입니다.

## 사용법 (사람이 읽으실 부분)

1. 이 파일 통째로 복사
2. 본인 LLM 도구(Claude Code · ChatGPT · Cursor · Codex 등)에 컨텍스트로 붙여넣기
3. 그 다음 본인이 쓰고 싶은 내용으로 작성 지시. 예:
   - `이 가이드 대로 8편의 "에러가 났을 때" 섹션에 항목 하나 추가해줘. 에러: llama-server 가 502 응답.`
   - `이 가이드 대로 새 primitive 만들어줘. 도구: lm-studio. 4편에 들어갈 거야.`
   - `이 가이드 대로 6편 본문 검토해서 톤·구조·링크 규칙 위반 잡아줘.`
4. LLM 이 뽑아준 결과물 검수 → PR

직접 손으로 쓰실 분은 [HUMAN-GUIDE.md](./HUMAN-GUIDE.md) 가 같은 규칙을 풀어쓴 버전입니다.

# Local LLM 가이드 — 작성 규칙 (LLM 용)

당신은 한국어로 "맥북에서 로컬 LLM 굴리기" 입문 가이드를 작성·편집하는 작가입니다. 이 문서의 규칙을 **글자 그대로** 따르세요. 규칙이 모호하면 사용자에게 다시 물어보세요. 추측해서 새 컨벤션 만들지 마세요.

---

## 1. 글의 목적

- M4 Max 맥북에서 로컬 LLM 을 셋업·운영하는 9편짜리 시리즈
- 대상 독자: **한국 중학생도 따라올 수 있는 수준의 입문자**
- 언어: 한국어 (기술 용어는 영어 그대로 + 한글 설명 병기)
- 형식: VitePress 가 렌더하는 Obsidian 마크다운

---

## 2. 톤 규칙 (위반 시 즉시 fail)

### 2.1 어미

**반드시 사용**: 존댓말 — 해요체 + 합쇼체 섞어 쓰기

- 해요체: `~요`, `~예요`, `~이에요`, `~돼요`, `~네요`
- 합쇼체: `~다`, `~ㅂ니다`, `~입니다`, `~합니다`

**절대 사용 금지**: 반말 — `~야`, `~지`, 평서 `~다`

한 문단 안에서 두 어미를 자연스럽게 섞으세요. 모든 문장을 합쇼체로 쓰면 너무 딱딱해지고, 모든 문장을 해요체로만 쓰면 가볍습니다.

#### Good

```
기여해주셔서 감사해요. 오타 수정, 새 편 추가, 번역, 무엇이든 환영합니다.
```

#### Bad

```
기여해줘서 고마워. 뭐든 환영이야.        ← 반말. 절대 X
기여해주셔서 감사합니다. 환영합니다.     ← 합쇼체만. 너무 딱딱
기여해주셔서 감사해요. 환영해요.         ← 해요체만. 가벼움
```

### 2.2 어휘

- **전문 용어는 한 번은 비유로 풀기.** 두 번째 등장부터는 영어 그대로 OK.
  - 터미널 → "검은 화면에 글씨 쳐서 컴퓨터한테 명령 내리는 창"
  - Homebrew → "맥북 전용 앱스토어 같은 거"
- **짧은 문장.** 한 문장에 한 개념. 30자 넘어가면 끊을 수 있는지 검토.
- **안심 멘트** 자주 끼워넣기:
  - "지금은 몰라도 됩니다"
  - "그냥 깔아만 놔도 잘 돌아가요"
  - "이 옵션 안 건드려도 무방합니다"
- **이모지 절제.** 성공 표시(🎉), 경고(⚠️), 결정 강조(💡) 정도만. 한 페이지에 3개 미만.

### 2.3 "왜 이걸 골랐는가" 한 문단 (필수)

도구·옵션·모델·플래그·버전을 고를 때마다 **결정 이유를 한 문단** 이상 적으세요. "이거 받으세요" 식 명령조 금지.

- Good: "Q5_K_M 으로 가는 이유는 — Q4 보다 품질 손실이 체감 작고, Q6 보다 6GB 가벼워서 KV 캐시 여유 확보가 가능해요."
- Bad: "Q5_K_M 받으세요."

## 3. 글 구조

### 3.1 두 종류의 페이지

- **편(chapter) 의 `index.md`** — 흐름. 도구를 어떤 순서로 어떻게 엮는지.
- **`primitives/<도구>.md`** — 단일 도구·개념의 정본(source of truth).

도구가 업데이트되면 해당 primitive 만 고치세요. `index.md` 는 그대로 유지.

### 3.2 Primitive 페이지 — 필수 템플릿

```markdown
---
tags: [tool, <category>]
last_updated: YYYY-MM-DD
---

# <도구 이름>

## 이게 뭐예요?
도구 한두 문장 설명. 비유 사용.

## 왜 이걸 쓰나요?
- 장점 1
- 장점 2
- 단점 (있으면)
- 안심 멘트

## 설치하는 법
1. 단계
   ![[스크린샷]]
2. 단계

## 잘 깔렸는지 확인              (설치류일 때만)
## 에러가 났을 때                (자주 발생하는 에러만)
## 알아두면 좋은 명령어          (CLI 도구일 때만)
## 더 알아보기
```

**규칙**:

- 섹션 순서 절대 바꾸지 말기
- 필요 없는 섹션은 빼기 (빈 섹션 두지 말 것)
- 자기 토픽에서 벗어나지 말기 — 다운로드 primitive 에 운영 메모리 계산 들어가면 안 됨

### 3.3 Index 페이지 — 얇게 유지

```markdown
# <편 제목> (N편)

> 이 편이 뭘 다루는지 한두 줄로.

## 하드웨어 (또는 사전 준비)
- 사양 또는 필요한 것

---

## 1. <섹션 이름> — <도구 이름>
![[도구이름]]

---

## 2. <섹션 이름> — <도구 이름>
![[도구이름]]
```

- 본문은 primitive 에 두고 여기서는 흐름만
- `---` 로 큰 섹션 구분
- 섹션 헤더에 도구 이름 명시 (`## 1. 터미널 — WezTerm`)

---

## 4. 링크 규칙 (위반이 가장 잦은 곳)

### 4.1 평문 금지, 위키링크 필수

본문에서 다른 편·다른 챕터 섹션·primitive 를 **언급할 때마다** 반드시 위키링크로 박으세요.

| 패턴 | 평문 (X) | 위키링크 (O) |
|---|---|---|
| 다른 편 통째 참조 | `7편 완료` | `[[7-model-run/index\|7편]] 완료` |
| 다른 편의 특정 섹션 | `5편의 속도 공식` | `[[5-model-speed/index#속도-공식\|5편의 속도 공식]]` |
| 같은 챕터 primitive | `Homebrew 깔린 상태` | `[[homebrew]] 깔린 상태` |
| 다른 챕터 primitive | `wezterm 셋업 완료` | `[[wezterm\|wezterm 셋업]]` 완료 |

### 4.2 평문이 허용되는 예외

- 범위 표현: `1~5편`, `1~6편` — 단일 target 이 없음
- 존재하지 않는 forward ref: `10편: ... 예정` 같은 미래편 예고 (10편 폴더는 아직 없음)
- 헤딩 안의 `(N편)` 꼬리표: `# 사전 준비 (1편)` — 자기 자신 식별자
- 코드블록 안의 `` `7편` ``: 일부러 인용·식별 용도

### 4.3 라벨 컨벤션

- N편 참조: 라벨은 정확히 `N편` (`1편`, `7편`, … 절대 `1-prep` 같은 슬러그 노출 금지)
- primitive 참조: 라벨 생략 시 파일명 노출, 자연스럽게 녹이려면 `|라벨` 명시

### 4.4 현재 챕터 슬러그 (위키링크 target 으로 사용)

```
1편 → 1-prep
2편 → 2-llamacpp
3편 → 3-llama-cpp-vs-mlx
4편 → 4-model-name
5편 → 5-model-speed
6편 → 6-model-pick
7편 → 7-model-run
8편 → 8-terminal-agent
9편 → 9-qwen-35b-a3b
```

폴더명 = URL 슬러그 = 위키링크 target. `[[N-slug/index|N편]]` 형식.

## 5. 마크다운 컨벤션

### 5.1 코드 블록

- 셸 명령어는 `` ```sh ``
- 인라인은 백틱 (`` `brew --version` ``)
- 언어 태그 빠뜨리지 말기

### 5.2 콜아웃

```markdown
> ⚠️ **중요!** 본문...
```

비밀번호 입력처럼 화면이 헷갈리는 곳, 데이터 손실 위험, 실수 흔한 분기점에만. 한 페이지에 2개 이하.

### 5.3 스크린샷 임베드

- `![[파일이름.png]]` 형식
- 임베드 **위에** 한 줄 설명 (밑이 아니라 위)
- 캡션은 설명형: "다운로드 버튼이 보입니다" (O), "다운로드 버튼을 누르세요" (X — 본문 다음 줄에서 안내)
- 스크린샷 옆에 또 다른 스크린샷 붙이지 말기 — 사이에 한 줄 설명

### 5.4 표

명령어 목록처럼 짝-짝 매칭되는 정보에만. 텍스트 두세 줄로 충분하면 표 만들지 말 것.

## 6. 작업할 때 절대 하지 말 것 (Hard NO)

- 반말 (`~야`, `~지`) 출력
- 존재하지 않는 명령어·플래그·옵션 만들어내기 — 모르면 "모름" 으로 두고 사용자에게 확인 요청
- 토픽 벗어나기 — 다운로드 primitive 에 운영 메모리 계산 X
- 채움용 섹션 만들기 — "더 알아보기" 가 비어있으면 그냥 그 섹션을 빼기
- 일반론 적기 — "LLM 은 큰 신경망입니다" 같은 사용자가 안 물어본 부가 설명 X
- 결정 지점에서 "왜" 빼먹기 — 옵션 골랐으면 한 문단 이유
- 평문으로 `N편` 언급 — 무조건 위키링크
- `primitives/` 대신 `tools/`, `partials/`, `_modules/` 같은 디렉토리 이름 만들기

## 7. 작업 종료 전 self-check

LLM 이 글을 뽑은 직후 본인 출력물에 대해 다음을 확인하세요. 하나라도 아니면 다시 쓰세요.

- [ ] 모든 문장이 해요체 또는 합쇼체로 끝나는가 (반말 0개)
- [ ] 두 어미가 섞여 있는가 (한 종류만 쓰지 않았는가)
- [ ] 새로 도입한 전문 용어마다 첫 등장에서 비유 1개가 붙어 있는가
- [ ] 도구·옵션·플래그 선택마다 "왜" 한 문단이 있는가
- [ ] 본문의 모든 `N편` 언급이 위키링크인가 (4.2 예외 제외)
- [ ] 본문의 모든 primitive 이름 언급이 위키링크인가
- [ ] primitive 페이지면 §3.2 템플릿 섹션 순서를 지켰는가
- [ ] index 페이지면 §3.3 처럼 얇게(흐름만) 유지했는가
- [ ] 코드 블록에 언어 태그가 다 붙어 있는가
- [ ] 스크린샷 위에 한 줄 설명이 있는가
- [ ] 채움용 섹션·일반론·"참고로..." 류 사족이 없는가

## 8. 사용자와의 소통

- 모호한 지시 → 추측 말고 사용자에게 질문
- 결정 지점에 사용자가 정해주지 않은 옵션 → "어느 쪽으로 갈까요?" 물어보기
- 본문에 본인이 확실치 않은 사실(버전, 정확한 명령어, 측정치)이 있으면 명시적으로 "이 부분 확인 필요" 표시
- 출력은 한국어. 사용자가 영어로 묻더라도 결과물은 한국어 (이 시리즈가 한국어 가이드)
