# Human Guide — 사람 손으로 글 쓰실 때

직접 키보드로 한 줄 한 줄 작성·편집하실 분을 위한 스타일 가이드입니다. LLM 에 던지실 거면 [LLM-GUIDE.md](./LLM-GUIDE.md) 를 보세요 — 같은 규칙을 LLM 친화 포맷으로 정리해뒀습니다.

---

## 1. 톤 & 보이스

### 존댓말 — 해요체 + 합쇼체 섞어 쓰기

|             | 어미 예시                               | 느낌                   |
| ----------- | --------------------------------------- | ---------------------- |
| 해요체      | `~요` `~예요` `~이에요` `~돼요` `~네요` | 부드럽고 친근          |
| 합쇼체      | `~다` `~ㅂ니다` `~입니다` `~합니다`     | 격식·문어체            |
| 반말 (금지) | `~야` `~지` 평서 `~다`                  | 친구·아랫사람 — 절대 X |

한 문단 안에서 두 어미를 자연스럽게 섞어 쓰면 "친구가 차근차근 설명해주는 선배" 톤이 나옵니다.

```text
기여해주셔서 감사해요. 오타 수정, 새 편 추가, 번역, 무엇이든 환영합니다.
       └─── 해요체 ───┘                              └ 합쇼체 ┘
```

### 한국 중학생도 이해할 수 있게

- **전문 용어는 비유로 풀기.** 예:
  - 터미널 → "검은 화면에 글씨 쳐서 컴퓨터한테 명령 내리는 창"
  - Homebrew → "맥북 전용 앱스토어 같은 거"
  - 패키지 매니저 → "도구를 명령어 한 줄로 깔아주는 도구"
- **짧은 문장**, 한 문장에 한 개념.
- **안심 멘트 자주**: "지금은 몰라도 됩니다", "그냥 깔아만 놔도 잘 돌아가니까 걱정 안 해도 돼요"
- **영어 기술 용어**(`brew`, `WezTerm`, `xcode-select` 등) 는 그대로 쓰되, 처음 등장할 때 한글로 짧게 설명 추가.
- **이모지는 절제해서.** 성공 표시(🎉), 경고(⚠️) 정도만.

### "왜 이걸 골랐는지" 한 문단

도구·옵션·모델·플래그를 고를 때마다 **결정 이유를 한 문단** 이라도 적어주세요. "이거 받으세요" 식 명령조 금지. 다른 선택지가 있는데 왜 이걸 골랐는지가 본문의 골자입니다.

## 2. 글 구조

이 시리즈는 두 종류의 페이지로 구성됩니다:

- **편(chapter) 의 `index.md`** — 흐름. 각 도구를 어떤 순서로 어떻게 엮는지.
- **`primitives/<도구>.md`** — 단일 도구·개념의 정본(source of truth).

도구가 업데이트되면 해당 primitive 만 고치면 돼요. `index.md` 는 그대로 유지.

### Primitive 페이지 템플릿

```markdown
---
tags: [tool, <category>] # terminal, package-manager, llm 등
last_updated: YYYY-MM-DD
---

# <도구 이름>

## 이게 뭐예요?

도구 한두 문장 설명. 비유 사용.

## 왜 이걸 쓰나요?

- 장점 1
- 장점 2
- 단점 (있으면)
- 안심 멘트

## 설치하는 법 (or 사용하는 법)

1. 단계
   ![[스크린샷]]
2. 단계
   ...

## 잘 깔렸는지 확인 # 설치류일 때만

명령어 + 결과 스크린샷

## 에러가 났을 때 # 자주 발생하는 에러만

### <에러 이름>

원인 + 해결 방법 단계별로

## 알아두면 좋은 명령어 # CLI 도구일 때만

| 명령어 | 무슨 뜻? |
| ------ | -------- |
| ...    | ...      |

## 더 알아보기

- 공식 문서 링크
- 설정 파일 위치 등
```

섹션은 **위 순서를 유지** 해주세요. 필요 없는 섹션은 그냥 빼면 됩니다 (빈 섹션 두지 마세요).

### Index 페이지 템플릿

```markdown
# <편 제목> (N편)

> 이 편이 뭘 다루는지 한두 줄로.

## 하드웨어 (또는 사전 준비)

- 사양 또는 필요한 것

---

## 1. <섹션 이름> — <도구 이름>

![[도구이름]]

---

## 2. <섹션 이름> — <도구 이름>

![[도구이름]]
```

- `index.md` 는 **얇게 유지** — 본문은 primitive 에 두고 여기서는 흐름만 잡습니다.
- `---` 로 큰 섹션 구분.
- 각 섹션 헤더에 도구 이름 같이 넣기 (예: `## 1. 터미널 — WezTerm`).

---

## 3. 마크다운 컨벤션

### 코드 블록

- 셸 명령어는 ` ```sh `
- 인라인은 백틱 (`` `brew --version` ``)

### 콜아웃 (중요/경고)

```markdown
> ⚠️ **중요!** 본문...
```

비밀번호 안 보이는 것 같은 헷갈리기 쉬운 부분, 데이터 손실 위험 등에만 사용. 남발 금지.

### 스크린샷 임베드

- `![[파일이름.jpg]]` 형식
- 임베드 **위에** 한 줄 설명 (밑이 아니라 위)
- 캡션은 설명형:
  - `다운로드 버튼이 보입니다` (O — 설명)
  - `다운로드 버튼을 누르세요` (X — 본문 다음 줄에서 안내)

### 표

명령어 목록처럼 짝-짝 매칭되는 정보에만. 텍스트로 충분하면 표 만들지 마세요.

### 링크 — 핵심 규칙

- 외부 URL: 그대로 노출 (`https://brew.sh/`)
- 내부 링크: `[[파일이름]]` 또는 `[[파일이름#섹션]]`
- 임베드: `![[파일이름]]`

#### 다른 편·primitive 언급은 **반드시** 위키링크 (평문 금지)

| 패턴                | 평문 (X)             | 위키링크 (O)                                         |
| ------------------- | -------------------- | ---------------------------------------------------- |
| 다른 편 통째 참조   | `7편 완료`           | `[[7-model-run/index\|7편]] 완료`                    |
| 다른 편의 특정 섹션 | `5편의 속도 공식`    | `[[5-model-speed/index#속도-공식\|5편의 속도 공식]]` |
| 같은 챕터 primitive | `Homebrew 깔린 상태` | `[[homebrew]] 깔린 상태`                             |
| 다른 챕터 primitive | `wezterm 셋업 완료`  | `[[wezterm\|wezterm 셋업]]` 완료                     |

**라벨 컨벤션**:

- N편 참조: 라벨은 `N편`
- primitive 참조: 자연스럽게 녹이려면 `|라벨` 명시

**평문 OK 예외**:

- 범위 표현: `1~5편`, `1~6편` — 단일 target 이 없어서 링크 안 됨
- 존재하지 않는 forward ref: `10편: ... 예정` 같은 미래편 예고
- 헤딩 안의 `(N편)` 꼬리표: `# 사전 준비 (1편)`
- 코드블록 안의 `` `7편` ``: 일부러 인용·식별 용도

#### 현재 챕터 슬러그

| #   | 슬러그               |
| --- | -------------------- |
| 1   | `1-prep`             |
| 2   | `2-llamacpp`         |
| 3   | `3-llama-cpp-vs-mlx` |
| 4   | `4-model-name`       |
| 5   | `5-model-speed`      |
| 6   | `6-model-pick`       |
| 7   | `7-model-run`        |
| 8   | `8-terminal-agent`   |
| 9   | `9-qwen-35b-a3b`     |

폴더명 = URL 슬러그 = 위키링크 target. `[[N-slug/index|N편]]` 형식으로 그대로 사용.

## 4. 자주 빠지는 함정

- **반말로 작성하지 않기.** 존댓말 유지.
- **새 primitive 만들 때 위 템플릿 따르기.** 임의로 섹션 순서 바꾸지 마세요.
- **`tools/`, `partials/`, `_modules/` 같은 이름 쓰지 말기.** 이 프로젝트는 `primitives/` 로 통일.
- **`index.md` 에 본문 길게 쓰지 말기.** 본문은 primitive 로 빼고 여기서는 embed 만.
- **스크린샷 옆에 또 다른 스크린샷 붙이지 말기.** 각 단계마다 한 줄 설명 끼워넣기.
- **존재하지 않는 명령어/플래그 만들어내지 말기.** 모르면 공식 문서 링크만 걸고 빈 자리로 둠.
- **토픽 벗어나지 말기.** 다운로드 primitive 에 운영 메모리 계산이 들어가지 않음. 인접 주제는 별도 primitive 에서 다루고 `![[...]]` 로 연결.
- **"왜 이걸 골랐는가" 빼먹지 말기.** §1 의 "결정 이유 한 문단" 규칙.
- **쓸데없는 정보 적지 않기.** "더 알아보기" 같은 채움용 섹션, 일반론, 독자가 안 물어본 부가 설명은 빼기.
- **다른 편·primitive 평문 언급 금지.** §3 → 링크 규칙 참고.
